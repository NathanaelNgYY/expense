# N1 — Pin "today/now" to SGT — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the client derive "today", "now", and new-entry dates from the Singapore (SGT) calendar date regardless of device timezone, matching the SGT server/ingest model.

**Architecture:** Add two date-granularity helpers to `src/shared/sgtDate.ts` (`sgtTodayString`, `sgtToday`) that convert the current instant to the SGT calendar date. Route every device-local `new Date()`-as-"now" and `toLocalDateString()`-as-"today" call site through them. `compute.ts` stays timezone-agnostic (it takes an injected `referenceDate`); the SGT responsibility lives at the screen boundary. Stored-entry bucketing (`entriesForMonth` reading `parseISO(date).getMonth()`) is already correct and is not touched.

**Tech Stack:** TypeScript, React 19, Vite, Vitest. Existing helper `sgtDateString` (Intl `en-CA`, `Asia/Singapore`) and `fromLocalDateString` (`src/dates.ts`).

## Global Constraints

- All date/budget logic is SGT-local via `src/shared/sgtDate.ts`; do not reintroduce raw device-local `Date` math for "now"/"today".
- `src/shared/` code is also consumed server-side — it must not import from `src/dates.ts` or any browser-only module.
- Helpers are **date-granularity only**; nothing here exposes SGT time-of-day.
- No storage, schema, or wire-format change. Existing entries and the server contract are untouched.
- The test suite pins no `TZ` (dev machine is SGT; CI is UTC). Deterministic tests either use absolute UTC instants or force `process.env.TZ` within the test.
- Do not weaken an existing assertion to make it pass; if a screen test implicitly assumed device-local "now", update it to the SGT expectation.

Append to `src/shared/sgtDate.ts`:

```ts
// The SGT (Asia/Singapore) calendar date `YYYY-MM-DD` for an instant.
export function sgtTodayString(now: Date = new Date()): string {
  return sgtDateString(now.toISOString())
}

// A Date whose LOCAL calendar fields equal the SGT calendar date (local-midnight of that day).
// Once a "now" value is sgtToday(), everything downstream — getMonth(), date-fns
// startOfWeek/endOfWeek, addDays, toLocalDateString — computes on the correct calendar date.
// Date-granularity only: time-of-day is midnight. Do not use this where SGT hour matters.
export function sgtToday(now: Date = new Date()): Date {
  const [year, month, day] = sgtTodayString(now).split('-').map(Number)
  return new Date(year, month - 1, day)
}
```

import { describe, it, expect, beforeAll, afterAll, vi } from 'vitest'
import { isFutureDateString } from './dates'

describe('isFutureDateString (SGT)', () => {
  const originalTz = process.env.TZ
  beforeAll(() => {
    // Force a non-SGT runner so a device-local implementation is provably wrong.
    process.env.TZ = 'UTC'
    vi.useFakeTimers()
    // 16:30Z => 2026-07-31 in UTC, but 2026-08-01 in SGT.
    vi.setSystemTime(new Date('2026-07-31T16:30:00Z'))
  })
  afterAll(() => {
    vi.useRealTimers()
    process.env.TZ = originalTz
  })

  it('treats the SGT calendar date as today, not the UTC date', () => {
    // Under a UTC device-local check this would be "future" (> 2026-07-31); under SGT it is today.
    expect(isFutureDateString('2026-08-01')).toBe(false)
  })
})
