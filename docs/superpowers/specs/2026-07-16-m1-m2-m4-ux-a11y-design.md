# M1 + M2 + M4 — lazy-load fallback, in-app confirm dialog, amount announcement

**Date:** 2026-07-16
**Status:** Approved design
**Source:** items M1, M2 and M4 of the July 2026 production/audit review (all MEDIUM), batched
because they are small, related perceived-quality + accessibility fixes, and all client-only.

## Problems

- **M1** — `App.tsx` uses `Suspense fallback={null}` in both boundaries (tab switches and
  first-run onboarding), so every lazy chunk load renders a blank screen.
- **M2** — Four destructive/discard flows use native `confirm()`: month reset
  (`src/screens/Settings.tsx`), unsaved-changes guard (`src/screens/settings/BudgetSettings.tsx`),
  delete shared budget (`src/sharedBudgets/OwnerTools.tsx`), leave shared budget
  (`src/sharedBudgets/BudgetDetail.tsx`). Native dialogs are unstyled, block the JS thread, and
  clash with the app's iOS-native feel.
- **M4** — The Add-entry amount display carries `aria-live="polite"` on the whole container
  (`src/screens/AddEntry.tsx`), so every numpad keypress re-announces the full amount string to
  screen readers.

## Design

### M1 — Delayed lazy-load fallback

New `src/components/LazyFallback.tsx`:

- Renders `null` for the first **150 ms** (timer + state), then a small centered spinner.
- Spinner is pure CSS animation using existing theme tokens; `role="status"` with accessible
  "Loading" text; `prefers-reduced-motion` swaps the spin for static "Loading…" text.
- Replaces `fallback={null}` in **both** Suspense boundaries in `App.tsx`.
- No per-screen skeletons. Chunks are small, so most loads finish inside the 150 ms window and
  the user still sees nothing — the delay exists precisely to avoid a spinner flash.

### M2 — `ConfirmDialog` + `useConfirm()`

New `src/components/ConfirmDialog.tsx` built on the **native `<dialog>` element with
`showModal()`** — free focus trap, Esc-to-cancel, top-layer stacking, no library.

Styling: iOS-style alert. Dimmed backdrop, centered card, title + message, two side-by-side
buttons — **Cancel** (initially focused; the safe default) and the action button, tinted with the
existing danger token when `destructive: true`.

API: promise-based hook via a `ConfirmProvider` mounted in `App`:

```tsx
const confirm = useConfirm()
// inside a handler:
if (!(await confirm({
  title: 'Delete 12 entries?',
  message: 'You can undo this while Settings remains open.',
  confirmLabel: 'Delete',
  destructive: true,
}))) return
```

Chosen over a declarative `<ConfirmDialog open …/>` per call site because all four existing sites
are inline `if (!confirm(…)) return` guards inside handlers; the promise API keeps the diff
minimal and reads the same way.

Migration — all four sites, no `window.confirm` left in `src/`:

| Site | Notes |
| --- | --- |
| `Settings.tsx` month reset | destructive, label "Delete" |
| `BudgetSettings.tsx` unsaved-changes guard | `handleBack` becomes async; non-destructive label "Leave" |
| `OwnerTools.tsx` delete shared budget | destructive, label "Delete" |
| `BudgetDetail.tsx` leave shared budget | destructive-adjacent, label "Leave" |

Cancel paths: Cancel button, Esc, backdrop tap — all resolve `false`.

### M4 — Debounced amount announcement

- Remove `aria-live="polite"` from `.amount-display`; keep its `aria-label` and the
  visually-hidden amount text so the value stays readable on focus/navigation.
- Add a separate **visually-hidden `role="status"` live region** in `AddEntry` that updates to
  the formatted amount **1 s after the last numpad press** (debounced; cleared on save and
  unmount). Typing is silent; the settled value is announced exactly once.

## Testing

TDD throughout; colocated Vitest tests.

- **M1:** fake-timer unit tests (nothing before 150 ms; spinner + status role after); App-level
  test that a lazy tab shows the fallback while its chunk is pending.
- **M2:** component tests — resolves true/false, Esc, backdrop, initial focus on Cancel,
  `role="dialog"` with accessible name. Update the five existing `window.confirm` spy tests
  (`BudgetSettings.test.tsx`, `BudgetDetail.test.tsx`, and the Settings/OwnerTools equivalents)
  to interact with the rendered dialog. CI Axe scans (e2e job) cover the dialog automatically.
  **Risk to verify first:** jsdom's `<dialog>`/`showModal()` support under our Vitest version —
  if absent, add a minimal polyfill in test setup, not a library.
- **M4:** fake-timer tests — no live-region update immediately on keypress; single announcement
  after the pause; no duplicate announcements on rapid input.

## Out of scope

- M3 (`role="tablist"` semantics / roving focus on the tab bar) — separate item.
- Any visual redesign of the screens themselves.
- Typed-confirmation ("type the budget name") hardening for shared-budget deletion — considered,
  rejected for now as friction disproportionate to a personal app.
