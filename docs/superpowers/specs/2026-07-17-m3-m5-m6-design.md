# M3 + M5 + M6 — Tab-bar semantics, desktop presentation, short screens

**Date:** 2026-07-17
**Status:** Approved
**Source items:** July 2026 audit — M3 (tab bar has button semantics, no focus management),
M5 (desktop and tablet are a 430px phone column), M6 (short-screen handling exists but is thin).

## Context

The five-tab bottom bar (`src/components/TabBar.tsx`) renders `<button aria-pressed>` inside
`<nav aria-label="Main navigation">`. There is no keyboard navigation between tabs beyond the
natural tab order. On viewports wider than 430px the app is a phone column centered on an
unstyled page background (`src/index.css` — `.app { max-width: 430px; margin: 0 auto }`).
Short-screen handling exists only for the Add screen in the original-dark theme
(`src/themes.css` — `@media (max-height: 720px)`); other screens and the other three themes
(Deep Sea Revised, Copper Current, Berry Circuit) have none.

## Decisions

Three scope decisions were made with the owner on 2026-07-17:

1. **M3 uses navigation semantics, not an ARIA tablist.** A bottom app bar is app-level
   navigation, not in-page panel switching. `role="tablist"`/`role="tab"` implies the
   tabs-and-tabpanel interaction pattern and is widely considered wrong for this control.
   The audit's literal wording ("tab semantics") is satisfied in intent: assistive technology
   announces the active destination and keyboard users get arrow-key movement.
2. **M5 keeps the single 430px column and makes it intentional.** No wider layouts, no
   multi-column screens. The product is an iPhone-first personal PWA; desktop is occasional.
3. **M6 is audit-driven.** Fix only observed defects at SE-class viewports; no speculative
   compact CSS. Add a regression guard so short-screen breakage cannot silently return.

## M3 — Tab bar: navigation semantics + keyboard support

`TabBar.tsx` keeps `<nav>` + five `<button>`s. Changes:

- The active tab exposes `aria-current="page"`; `aria-pressed` is removed. Inactive tabs have
  no `aria-current` attribute (not `aria-current="false"`).
- Keyboard: `ArrowLeft`/`ArrowRight` on any tab button moves focus to the previous/next button,
  wrapping at the ends. `Home`/`End` move focus to the first/last button. Arrow keys move
  **focus only** — they do not activate the tab (activation stays on Enter/Space/click, matching
  the disclosure-nav pattern rather than automatic-activation tabs).
- All five buttons remain in the natural tab order (no roving `tabindex`). Arrow keys are a
  progressive enhancement; nothing regresses for users who only use Tab.
- Visual `.active` class logic and `onChange` behavior are unchanged.

Knock-on updates:

- `tests/e2e/journeys.spec.ts` asserts `aria-pressed` on the Settings tab (five-tab navigation
  journey) — assert `aria-current="page"` instead.
- Any unit tests asserting `aria-pressed` on tab buttons follow the same swap.

## M5 — Desktop/tablet: polished intentional column

At `@media (min-width: 700px)` only:

- The page background behind the column (on `body` or `#root`) becomes a themed backdrop
  derived from each theme's existing tokens with `color-mix()` — a quiet tone shift, not a new
  palette. All four themes are covered by one rule referencing theme CSS variables; no
  per-theme hex values.
- `.app` gains a soft elevation shadow and a hairline border (`1px` at reduced opacity from a
  theme token) so the column reads as a deliberate app surface.
- Nothing inside the column changes: no width, spacing, or component changes at any viewport.
  Mobile (< 700px) renders byte-identical CSS outcomes to today.

## M6 — Short screens: audit-driven fixes + regression guard

1. **Audit.** Drive every primary screen — Home, History, Add, Insights, Settings hub, and the
   Settings subscreens (Automatic Tracking, Budget & Categories, Appearance, Data & Backup) —
   at **320×568** and **375×667**, across all four themes, using Playwright screenshots.
   The audit produces a written defect list (screen × theme × viewport × symptom).
2. **Fix.** Only observed defects qualify: clipped or hidden primary actions, content trapped
   under the tab bar, horizontal overflow, control heights below 44px. The existing original-dark
   Add-screen compact treatment is the stylistic template when the same failure appears in
   other themes. If a screen shows no defect, it gets no CSS.
3. **Guard.** A new e2e check at **320×568** asserts, for each audited primary screen:
   no horizontal overflow (`scrollWidth <= innerWidth`) and the screen's primary action
   (e.g. Save on Add; the Reset button on Settings after scrolling) can be scrolled into
   view and measures ≥ 44px tall. This extends the existing pattern already used at
   375×667 and 390×844.

## Error handling

No new runtime code paths: M3 adds a keyboard handler (pure focus movement, no async), M5/M6
are CSS-only. The keyboard handler ignores keys it does not own so screen-reader/browser
default behavior is never blocked.

## Testing & gates

- **Unit:** TabBar tests — active button exposes `aria-current="page"`, inactive buttons do
  not; ArrowRight from the last tab wraps to the first; ArrowLeft from the first wraps to the
  last; Home/End land on first/last; arrow movement does not fire `onChange`.
- **E2E:** update the `aria-pressed` assertion; add the 320×568 short-viewport guard; the
  existing axe WCAG A/AA scans re-run unchanged.
- **Bundle:** the CSS budget is 13 KiB gzip (12.3 actual). M5/M6 CSS should fit inside the
  0.7 KiB headroom; if the gate trips, escalate to the owner (per the M1/M2/M4 precedent) —
  never raise the budget silently.

## Out of scope

- Wider desktop layouts, sidebars, or multi-column screens (explicitly rejected for M5).
- ARIA tablist/tabpanel semantics (explicitly rejected for M3).
- Speculative compact CSS for screens with no observed short-viewport defect (M6).
- Any change to screen content, business logic, or state management.
